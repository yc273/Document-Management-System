# 功能实施方案：文件夹软删除 + 分片上传

## 功能一：文件夹软删除（回收站支持文件夹）

### 后端改动

**1. `backend/app/models/folder.py`** — 加字段+方法
- 加两个字段：`is_deleted` (Integer, default=0, indexed) 和 `deleted_at` (DateTime, nullable)
- 加 `soft_delete()` / `restore()` 方法（镜像 File 模型）
- 加 `get_descendant_folder_ids()` 方法：递归收集所有子孙文件夹ID（用于级联软删除/恢复）
- 修改 `to_dict()`：is_deleted=1 时附加 `deleted_at` 和 `expire_days`（复用 TRASH_RETENTION_DAYS）
- 修改 `get_folder_tree()`、`get_root_folders()`、`get_children()`、`get_file_count()`：都加 `is_deleted == 0` 过滤
- 删除 `can_delete()` 的拦截逻辑（改为允许级联软删除）

**2. `backend/app/controllers/folder_controller.py`**
- `delete_folder()`：改为级联软删除——收集 folder + 所有子孙 folders，连同其中的 files 一起标记 is_deleted=1 + deleted_at
- `create_folder()` 的重名检查：加 `is_deleted == 0` 过滤（软删除的文件夹名可复用）
- `get_folder_list()`：已用 get_folder_tree（会自动过滤）

**3. `backend/app/controllers/trash_controller.py`**
- `get_trash_list()`：同时返回软删除的文件夹（带 `item_type: 'folder'/'file'` 区分）
- 新增 `POST /api/trash/restore-folder/<folder_id>`：整体恢复——递归恢复文件夹+子孙文件夹+内部文件；若父文件夹也软删除则一起恢复；若父文件夹不存在则挂到根目录
- 新增 `DELETE /api/trash/delete-folder/<folder_id>`：永久删除——递归删除文件夹+子孙+内部文件（调用 `_safe_remove_physical_file`）
- `auto_cleanup_expired_trash()`：扩展，同时清理过期的软删除文件夹（级联）
- `clear_trash()`：扩展，同时清空回收站中的文件夹

**4. `scripts/init_db.sql`** — 文档同步：doc_folder 表加 is_deleted/deleted_at 字段

**5. 数据库迁移** — 对现有库执行 `ALTER TABLE doc_folder ADD COLUMN is_deleted INTEGER DEFAULT 0; ALTER TABLE doc_folder ADD COLUMN deleted_at DATETIME;`（db.create_all() 不会给已存在的表加列）

### 前端改动

**6. `frontend/src/api/trash.js`** — 加 `restoreFolder(id)`、`permanentDeleteFolder(id)`

**7. `frontend/src/views/Trash.vue`**
- 表格根据 `item_type` 渲染：文件夹行用 Folder 图标，隐藏大小列（或显示「—」）
- 恢复/删除操作根据 item_type 调用对应 API
- 批量恢复/删除要同时处理 file 和 folder 两种类型

## 功能二：分片上传（大文件自动切换）

### 策略
前端文件 < 20MB 走原有单次上传；≥ 20MB 自动切换分片上传（每片 5MB），支持暂停/续传/进度显示。

### 后端改动

**8. `backend/app/controllers/file_controller.py`** — 新增3个接口
- `POST /api/file/upload/check`：秒传+断点续传检测。前端先计算文件 MD5，传 file_hash + file_size + filename + folder_id。返回：已存在(秒传) / 已上传分片列表(断点续传) / 全新文件
- `POST /api/file/upload/chunk`：接收单个分片。参数：file_hash, chunk_index, total_chunks, chunk(blob), filename, folder_id, total_size。存到 `UPLOAD_TEMP/{file_hash}/chunk_{index}`
- `POST /api/file/upload/merge`：合并分片。参数：file_hash, filename, folder_id, total_size。合并所有分片 → 移到 UPLOAD_DOCUMENTS → 走已有的去重 + create_file 流程 → 清理临时分片

**9. `backend/app/services/upload_service.py`** — 加分片合并逻辑
- `merge_chunks(file_hash, filename, user_id, folder_id, total_size)`：合并文件、校验大小、复用 handle_duplicate_file 做秒传判断、创建 File 记录

**10. `backend/app/models/file.py`** — `create_file()` 加可选 `file_hash` 参数，避免合并后重复算 MD5

### 前端改动

**11. `frontend/src/api/file.js`** — 加 `checkUpload(data)`、`uploadChunk(formData, onProgress)`、`mergeUpload(data)`

**12. `frontend/src/views/FileManager.vue`**
- `beforeUpload`：移除 100MB 限制（改为后端配额控制）
- 新增分片上传逻辑：文件 ≥20MB 时，用 `Blob.slice` 切片，先调 check 接口（秒传/续传判断），再并发上传分片（限制3并发），最后调 merge
- 上传对话框加进度条（显示总进度 + 单文件状态）、暂停/继续按钮

**13. `frontend/src/utils/request.js`** — 确认 blob 响应放行（已改过）

## 实施顺序
1. 先做**文件夹软删除**（影响面大，先把数据层理清）
2. 再做**分片上传**（相对独立）

每个功能完成后做语法检查 + 数据库验证，最后重启后端联调。
